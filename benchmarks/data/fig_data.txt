S2.1.3.7. Decoupled Objective Optimization with Multi-Objective TradeoW Mapping 
And ﬁnally, for case study 8 involving the chiral 2+2 cycloaddition ,(Case study 6, Section S3.7), a 
hybrid strategy was employed to balance the use of batching and human-in-the-loop 
methodology without compromising the ediciency of the optimization. In this approach, 
suggested experimental points were selected by maximizing the output of individual Gaussian 
processes trained separately for each target value. This means that one experiment was 
proposed per target, each chosen to independently optimize diastereomeric ratio, yield, or 
enantioselectivity. To complement these, a set of additional points was selected using a Pareto-
based approach, speciﬁcally the qLogNParEGO algorithm,44,45 in order to map out the trade-od 
space between the multiple objectives. This strategy provides the experimentalist with granular 
control over which tradeod between objectives is acceptable when deciding on the conditions to 
carry forward into scale-up.21 It emphasizes that even when high-throughput or fully in-line 
analytics are unavailable, automated experimentation can remain an edective and ﬂexible tool 
when combined with intelligent batching and clear prioritization of objectives.

S2.2. 
Benchmarking 
Benchmarking is essential as a smoke test for the Bayesian optimization methodologies 
employed across campaigns. Detailed benchmarking results are provided in the respective 
campaign sections; this section outlines the overarching benchmarking framework. Benchmarks 
were performed using datasets from both the Olympus package46 (developed by the Aspuru-
Guzik group) and the Minerva package (by Schwaller and co-workers), covering a wide range of 
realistic chemical optimization tasks. The benchmarking architecture follows an oracle-based 
model: each BO strategy queries a ground-truth model, trained on a literature database. This 
oracle framework enables rapid and repeatable assessment of performance. To accelerate 
testing, the entire pipeline is parallelized using a multiprocessing approach, allowing large 
sweeps of parameter conﬁgurations to be tested in a fraction of the time. A full factorial design is 
used to systematically explore the space of selectable options, such as surrogate model, 
acquisition function, and encoding strategy, enabling correlation analyses between these 
choices and the observed optimization performance. 
S2.2.1. 
Choice of data and Surrogate models 
The benchmarking data was selected to closely mimic the type of data expected from real 
experimental campaigns. To that end, three key criteria were used: (1) the dataset must contain 
chemical reaction data, (2) it must include at least one categorical variable, and (3) it must feature 
at least two optimization targets. Among the available options, datasets related to Suzuki–
Miyaura couplings in the Olympus package46 met all criteria. After initial exploratory analysis, 
Suzuki I and Suzuki IV were selected as the primary benchmarking datasets. A variety of surrogate 
models were evaluated for use as the oracle in the benchmarking framework. These included 
Random Forest Regressor, Gaussian Process, Multi-linear Regression, Ridge Regression, and 
second-order polynomial-corrected Linear Regression27 To better reﬂect real-world BO behavior, 
and to improve surrogate accuracy, the data was subdivided by unique combinations of 
categorical variables. Separate surrogate models were then trained on each continuous 
subspace, using min-max scaling to normalize all features to the [0,1] range.47 Model evaluation 
revealed that both Random Forest and Gaussian Process regressors performed well in terms of 
predictive accuracy.  However, due to the slight better performance exhibited by the RF models 
they were chosen as the oracle models.48

S2.2.2. 
Emulator Engine 
The emulator engine is designed to simulate experimental outcomes by interfacing with pre-
trained surrogate models. It takes as input the name of the dataset to emulate and the choice of 
oracle model. The engine begins by splitting the dataset into sub-datasets based on predeﬁned 
categorical columns, edectively isolating regions of the design space with consistent categorical 
settings. A separate surrogate model is then trained for each sub-dataset and stored in memory 
for edicient querying. The user can also specify a desired noise level to simulate experimental 
uncertainty. During evaluation, the Bayesian optimizer submits a structured input containing one 
or more 1-D tensors representing the design points to be evaluated. The emulator engine 
automatically identiﬁes the categorical values in each input, routes the request to the 
corresponding surrogate model, and returns the predicted result. If a noise level has been 
speciﬁed, the emulator applies normally sampled noise to the result. As the normally sampled 
noise is additive the optimum of that calculation is higher than with no noise. 
S2.2.3. 
Benchmarking engine 
The benchmarking engine operates from a YAML input ﬁle, where each benchmark conﬁguration 
is deﬁned. It constructs the full benchmarking space by computing the Cartesian product of all 
speciﬁed parameters, ensuring comprehensive coverage of possible settings. Each conﬁguration 
is submitted as a task to a multiprocessing queue, where it is processed independently to 
maximize ediciency. Each benchmark run uses the same modular communication infrastructure 
as the robotic experimentation system. Upon execution, the benchmarking process loads the 
speciﬁed emulator and BO methodology, then proceeds with the optimization. Outputs are 
recorded in a structured format: results are saved as CSV ﬁles, while the corresponding input 
parameters are stored in a JSON ﬁle to ensure traceability and reproducibility. Additionally, the 
benchmarking engine manages run-time notiﬁcations, alerting the user when a benchmark starts 
and ﬁnishes. To support robustness during long or interrupted campaigns, the engine also 
includes a restart-aware feature that skips previously completed runs, allowing seamless 
continuation of benchmarking without duplication.

S2.3. 
Further Data Analysis 
Each campaign’s data is further analyzed using standard statistical techniques. While the 
datasets generated through Bayesian optimization may be biased toward exploitation of speciﬁc 
regions of the design space, it is not yet clear to what extent this bias adects the validity of 
downstream statistical analyses. Nevertheless, forgoing further analysis would represent a 
missed opportunity to extract deeper insights into the underlying chemical systems. As with the 
previous sections, this serves as an introduction; detailed data analyses are presented in the 
corresponding subsections of each campaign. Below is a detailed list of techniques applied and 
what they seek to showcase. 
S2.3.1. 
Targets Over Experiment Progress 
Scatter plots of the tracked objectives and the hypervolume, calculated as the area of the non-
dominated partitioning using 0n as the reference point, are included to visualize the progression 
of each optimization campaign. These plots provide an overview of how the model explores and 
exploits the design space over time, odering insight into whether the surrogate model is 
edectively learning and navigating the underlying chemical system. 
S2.3.2. 
Correlational Analysis of Input Features and Targets 
Although correlational analysis of the input space may not always yield mechanistically 
meaningful interpretations, particularly in a dataset biased by optimization, it still oders valuable 
insight into how the optimizer perceived interactions between features. This is especially relevant 
for Gaussian Process (GP) models, where the kernel relies on assumptions about correlation 
structure in the input space.49 For continuous variables, both Pearson’s R and Spearman’s rank 
correlation coedicients are calculated to assess linear and monotonic relationships, 
respectively.50 For categorical feature, we treat each category as a separate dummy variable and 
compute edect sizes using Eta squared (η²) and Omega squared (ω²) to quantify the variance in 
the objective explained by each categorical value.51 This multi-level analysis helps identify which 
features, whether continuous or categorical, had the largest inﬂuence on the optimization 
outcome, and reveals interactions between categorical inputs that may have guided the 
optimizer’s selections.
